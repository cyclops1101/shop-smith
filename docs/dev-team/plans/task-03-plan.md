# Task 03 Plan: Database Migrations — All 17 App Tables

**Task ID:** 03
**Domain:** backend
**Parallel Group:** 2 (depends on Task 01)
**Complexity:** high
**Status:** pending

---

## 1. Approach

Create all 18 migration files (17 application tables + 1 polymorphic pivot) that form the complete database schema for the Workshop Manager. These run on top of the 3 existing Laravel baseline migrations (users, cache, jobs) and must not touch them.

The core challenge is ordering: MySQL enforces foreign key constraints at migration time, so each migration must run after all tables it references. The approach is to generate migration files using `sail artisan make:migration` in strict dependency order — each command runs in sequence so that the auto-generated timestamps increment, guaranteeing the correct execution order when `php artisan migrate` runs.

After all files are generated, each migration's `up()` method is filled in with the exact column definitions from the spec. All `down()` methods use `Schema::dropIfExists()`. The implementation is then verified by running `php artisan migrate:fresh` against the MySQL 8 database.

**What is NOT done in this task:**
- Eloquent models (Task 04)
- Factories or seeders (Task 08)
- Enum PHP classes (Task 02) — enum values are stored as plain strings in the DB; no PHP enum files are needed to create the schema

## 2. Migration Ordering — Dependency Graph

Tables must be created in an order where no table references a not-yet-existing table via foreign key. The dependency order (and thus the order to run `make:migration`) is:

```
Level 0 (no FK deps):
  tags
  material_categories
  tool_categories
  suppliers
  projects               (soft deletes)

Level 1 (depends on Level 0):
  project_photos         -> projects (CASCADE DELETE)
  project_notes          -> projects (CASCADE DELETE)
  time_entries           -> projects (CASCADE DELETE)
  materials              -> material_categories (SET NULL), suppliers (SET NULL)
  tools                  -> tool_categories (SET NULL)
  expenses               -> projects (SET NULL), suppliers (SET NULL)
  revenues               -> projects (SET NULL)
  cut_list_pieces        -> projects (CASCADE DELETE)

Level 2 (depends on Level 1):
  project_materials      -> projects (CASCADE DELETE), materials (CASCADE DELETE)
  maintenance_schedules  -> tools (CASCADE DELETE)
  cut_list_boards        -> projects (SET NULL), materials (SET NULL)

Level 3 (depends on Level 2):
  maintenance_logs       -> tools (CASCADE DELETE), maintenance_schedules (SET NULL)

Pivot (depends on tags):
  taggables              -> tags (CASCADE DELETE)
```

Note: `expenses` and `revenues` depend on both `projects` and `suppliers` (both Level 0), so they can be created at Level 1. `cut_list_boards` depends on both `projects` (Level 0) and `materials` (Level 1), placing it at Level 2.

## 5. Detailed Migration Specifications

### ULID Primary Key Convention

Every application table uses:
```php
$table->ulid('id')->primary();
```

This creates a `char(26)` NOT NULL column that MySQL uses as the primary key. Laravel's `HasUlids` trait (applied in Task 04) populates this automatically.

**Foreign key ULID columns** use `foreignUlid()`:
```php
$table->foreignUlid('project_id')->constrained('projects')->cascadeOnDelete();
```

**Nullable foreign key ULID columns:**
```php
$table->foreignUlid('category_id')->nullable()->constrained('material_categories')->nullOnDelete();
```

### Cascade Rule Reference

| Relationship | Cascade Behavior | Rationale |
|---|---|---|
| project_photos → projects | CASCADE DELETE | Photos have no meaning without their project |
| project_notes → projects | CASCADE DELETE | Notes are project-scoped |
| time_entries → projects | CASCADE DELETE | Time entries are project-scoped |
| materials → material_categories | SET NULL | Preserve material inventory if category removed |
| materials → suppliers | SET NULL | Preserve material inventory if supplier removed |
| project_materials → projects | CASCADE DELETE | Pivot record meaningless without project |
| project_materials → materials | CASCADE DELETE | Pivot record meaningless without material |
| tools → tool_categories | SET NULL | Preserve tool records if category removed |
| maintenance_schedules → tools | CASCADE DELETE | Schedules are tool-scoped |
| maintenance_logs → tools | CASCADE DELETE | Logs are tool-scoped |
| maintenance_logs → maintenance_schedules | SET NULL | Preserve log history if schedule deleted |
| expenses → projects | SET NULL | Financial records have standalone value |
| expenses → suppliers | SET NULL | Financial records have standalone value |
| revenues → projects | SET NULL | Financial records have standalone value |
| cut_list_boards → projects | SET NULL | Boards can be general stock (null project) |
| cut_list_boards → materials | SET NULL | Preserve board if material record removed |
| cut_list_pieces → projects | CASCADE DELETE | Pieces are always project-scoped |
| taggables → tags | CASCADE DELETE | Tag assignments are meaningless without the tag |

## 6. Key Decisions

### Decision 1: `foreignUlid()` — not `ulid()` plus a separate `foreign()` call

Laravel's `$table->foreignUlid('column')` creates a `char(26)` column and supports chained `->constrained()`, `->cascadeOnDelete()`, and `->nullOnDelete()`. This is the clean, idiomatic way to define ULID foreign keys in Laravel 10+. The alternative (manually calling `$table->ulid('column')` then `$table->foreign('column')->references('id')->on('table')`) is more verbose and error-prone.

### Decision 2: Enum values stored as `string`, not MySQL `ENUM` type

All status/type columns use `$table->string('status', 30)` (or similar length), not `$table->enum('status', [...])`. Reasons:
- MySQL `ENUM` type requires a full table `ALTER` to add a new value — string columns do not
- PHP backed enums handle validation at the application layer; the DB stores the raw string
- String columns are portable and inspectable
- CLAUDE.md explicitly states "PHP Enums (backed by strings) for status/type fields"

### Decision 3: Timestamps only where spec specifies `updated_at`

The spec includes explicit column lists per table. Tables with only `created_at` (not `updated_at`): `project_photos`, `time_entries`, `project_materials`, `maintenance_logs`, `cut_list_boards`, `cut_list_pieces`. These use `$table->timestamp('created_at')->nullable()` rather than `$table->timestamps()`. The corresponding Eloquent models (Task 04) must configure `public const UPDATED_AT = null;` to prevent Eloquent from trying to set `updated_at` on a column that does not exist.

Tables with no timestamps at all (per spec): `tags`, `material_categories`, `tool_categories`, `taggables`. These use neither `$table->timestamps()` nor a single `created_at` column.

### Decision 4: No `id` on `taggables` pivot

The polymorphic pivot `taggables` uses a three-column composite primary key (`tag_id`, `taggable_id`, `taggable_type`). No auto-increment `id` is needed. Laravel's `morphToMany` relationship uses `attach()`/`detach()` methods that do not require a surrogate key on the pivot.

### Decision 5: `project_materials` has its own ULID `id`

Although `project_materials` is conceptually a pivot table, the spec gives it extra columns (`quantity_used`, `cost_at_time`, `notes`) and a `created_at` timestamp. It gets a ULID primary key (`id`) so records can be individually referenced (e.g., to update quantity or delete a specific material allocation). The unique composite index `(project_id, material_id)` enforces that the same material appears at most once per project.

### Decision 6: `make:migration` generates timestamps in execution order

Running `make:migration` commands sequentially produces files named like:
```
2026_03_03_100001_create_tags_table.php
2026_03_03_100002_create_material_categories_table.php
...
```

Laravel sorts migrations by filename before running them. As long as the commands are issued in the dependency order listed in Section 2, the timestamps will be in the correct order. This is more reliable than manually setting timestamps in filenames.

### Decision 7: `nullOnDelete()` on nullable FKs, `cascadeOnDelete()` on required FKs

This matches both the spec's intent and the cascade rule reference table. Financial records (`expenses`, `revenues`) are preserved when their associated project or supplier is deleted (SET NULL) because they have accounting value. Structural records (`project_photos`, `time_entries`, `maintenance_logs`) are cascade-deleted because they have no meaning without their parent.

## 8. Risks and Mitigations

### Risk 1: FK constraint errors from wrong migration order

**Risk:** If `create_project_materials_table` runs before `create_materials_table`, MySQL throws `Cannot add foreign key constraint`.

**Mitigation:** Commands in Section 4 are listed in strict dependency order. Run them exactly in that sequence. Before running `migrate:fresh`, verify the migration filenames have incrementing timestamps matching the order in Section 3. If any file has an out-of-order timestamp, rename it manually.

### Risk 2: `project_materials` unique constraint may block future features

**Risk:** The unique composite index `(project_id, material_id)` prevents attaching the same material to a project twice. If a future feature requires tracking material usage across multiple sessions (e.g., two separate purchase orders of the same material for one project), this constraint would block it.

**Mitigation:** The constraint matches the spec exactly. If this requirement changes, it is a simple `ALTER TABLE DROP INDEX` migration. No action needed now.

### Risk 3: Tables without `updated_at` cause Eloquent `save()` failures

**Risk:** For tables with `created_at` only (not `updated_at`), if the Eloquent model has `$timestamps = true` (the default), calling `$model->save()` on an existing record will attempt to write `updated_at` and fail with a column-not-found error.

**Mitigation:** This is a Task 04 concern. Each affected model must set `public const UPDATED_AT = null;`. The migration itself is correct; the schema only has the columns the spec defines. Document the list of affected models clearly for the Task 04 implementing agent: `ProjectPhoto`, `TimeEntry`, `ProjectMaterial`, `MaintenanceLog`, `CutListBoard`, `CutListPiece`.

### Risk 4: `taggables` composite primary key and Eloquent morphToMany

**Risk:** Laravel's `morphToMany` relationship expects the pivot to be queryable by `taggable_id` + `taggable_type`. With a composite primary key and no auto-increment `id`, the attach/detach operations must work through the composite key.

**Mitigation:** Laravel's `morphToMany` and the `taggables` table structure with a composite primary key is the standard pattern for polymorphic many-to-many in Laravel. The `index(['taggable_id', 'taggable_type'])` added in the migration ensures these queries are fast. No additional configuration is needed beyond what is defined here.

### Risk 5: MySQL 8 ULID column size check

**Risk:** `$table->ulid()` creates `char(26)` in MySQL 8. The acceptance criterion mentions "varchar(26)" but MySQL uses `char(26)` for ULIDs. A verification step looking for `varchar(26)` would incorrectly fail.

**Mitigation:** The acceptance criterion wording is imprecise. The verification step must look for `char(26)`, not `varchar(26)`. When running `SHOW COLUMNS FROM projects WHERE Field = 'id'`, the expected output is `char(26) NOT NULL`. This is correct behavior and not a schema error.
